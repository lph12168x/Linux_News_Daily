NVK 开源驱动首次支持 DLSS：Mesa 26.2-devel 整合实验性实现 GPU 计算
6月20日·Phoronix / TechPowerUp / sina / freedesktop GitLab

Mesa 26.2 开发分支正式合入一项里程碑式更新：开源 NVIDIA NVK Vulkan 驱动首次获得 DLSS（深度学习超采样）支持。

核心要点：

  • 实现路径 — 通过 VK_NVX_binary_import Vulkan 扩展，NVK 驱动加载并执行 DLSS 预编译二进制，绕开 NVIDIA 专有闭源代码
  • 贡献来源 — 补丁源自 Valve 开发者 Autumn Ashton 去年提交的工作，Collabora、Red Hat、Valve 共同推进
  • 默认实验性 — 用户需设置环境变量 NVK_EXPERIMENTAL=dlss 显式启用，预计 8 月 Mesa 26.2 稳定版发布时随之上线
  • 性能差距仍在 — 近期 Phoronix 基准显示 NVK 与 NVIDIA 闭源驱动仍存在 30-40% 性能差，DLSS 支持可部分缓解此差距
DLSS 是 NVIDIA 在 RTX 平台上推广 AI 升频的核心卖点，过去完全依赖闭源驱动。NVK 获得 DLSS 支持意味着：① Steam Deck、Linux 游戏玩家不再需要"DLSS 二选一" — 开源栈也能跑 AI 升频；② 配合 NVK 持续推进的功能对等（光追、网格着色器），Linux NVIDIA 显卡的体验正在快速逼近 Windows；③ Valve/Collabora 主导的开源 NVK 已具备冲击企业级工作站市场的潜力，对 NPU/GPU 生态格局有深远影响。

Linux 7.2 合并窗口引入 Cache Aware Scheduling：现代多 LLC CPU 性能突破 内核 计算
6月19日·LWN.net / Phoronix / charlie27

排程子系统合入了醞酿逾一年、由 Intel 工程师主导开发的 Cache Aware Scheduling（CAS），由维护者 Peter Zijlstra 通过 sched/cache 分支合入主線：

  • 核心机制 — 任务迁移决策时感知末层缓存（LLC）边界，将频繁共享数据的执行緒固定于同一 LLC domain，降低跨域一致性流量与互连延迟
  • 运行时切换 — 通过 CONFIG_SCHED_CACHE（默认关闭）控制编译期启用，提供 debugfs 开关 /sys/kernel/debug/llc_balancing/enabled 无需重启做 A/B 测试
  • 主要受益场景 — AMD EPYC Turin（多 CCD 各持独立 L3）、Intel Xeon 6 Granite Rapids（多 tile 独立 LLC）、ARM 多集群服务器；MPI、数据库线程、DPDK、AI 推理后端 性能提升明显
  • 同批合入 — sched_ext cgroup 子调度器（每 cgroup 挂载独立 BPF 调度逻辑）、SD_ASYM_CPUCAPACITY SMT 感知优化、cfs_rq 数据局部性分配

初步基准显示 AMD EPYC Turin 性能改善最显著，Intel Xeon 6 同样有正向增益，且未观察到主要调度回归。合并窗口预计 6 月 28 日关闭，7.2-rc1 释出，正式版 8 月中下旬发布。

这是 2023 年 EEVDF 替代 CFS 之后，Linux 调度器最重要的架构升级。对图形/计算领域而言：① NPU/GPU 后端调度（如 vLLM、TensorRT-LLM）大量依赖 CPU 线程与 GPU 协作，CAS 可显著降低 PCIe/NVLink 通信的 CPU 侧等待；② 多卡训练/推理时，CPU-GPU 绑定关系更加稳定，K8s 上的 device plugin 调度器也能更好配合；③ 对运行模拟、渲染流水线的 NUMA 敏感工作负载（如 Blender Cycles、Vulkan Ray Tracing）是直接利好。

RMR/BRMR：单跳 RDMA 多播为云端持久磁碟提供新方案 计算 内核
6月19日·LWN.net / LKML / kernel.org

在 2026 年 LSFMM+BPF 峰会上，开发者 Jia Li 与合作者发布两个新核心模块：RMR（Reliable Multicast over RTRS） 与 BRMR（Block device over RMR）：

  • 解决痛点 — 裸 RDMA 不提供块设备所需的持久性与排序保证。传统方案（如 DRBD）采用双跳：写入先落主端本地磁盘，再同步至副本，两段延迟叠加
  • 单跳设计 — RMR 在 RTRS 基础上加入可靠多播语义，一次写入通过 RDMA 同时扇出至多个副本节点，所有副本 ACK 后方视完成，实现原子性扇出。理论端到端写延迟可减半
  • 接口设计 — BRMR 在 RMR 之上提供标准 Linux 块设备接口（struct block_device_operations），用户态可通过 /dev/brmrX 访问
  • 与现有栈的关系 — 与 RNBD/RTRS（5.14 已合入）共存，RTRS 作为共享传输层，定位"需要跨多节点同步复制的高可用虚拟磁盘"，对标 AWS EBS / GCP Persistent Disk

目前仍处 RFC 与 PoC 阶段（patch series 00/13），尚未合入主线。峰会的目的是收集核心存储与 RDMA 维护者早期反馈。

对关注 RDMA 的开发者来说，RMR/BRMR 的意义在于：① 给出了一条与 Ceph/iSCSI 完全不同的"内核态低延迟块复制"路径，避免了 userspace 代理的额外开销；② "单跳"是 RDMA 一直追求但难以达到的目标 — 该方案展示了 RDMA 多播语义可以弥补块设备的持久性短板；③ 对云数据库、分布式存储、AI 训练数据湖（需要跨 AZ 复制）等场景意义重大。对 NPU/RDMA 工作背景的开发者，这是值得持续追踪的方向 — 它的设计模式可能影响未来 Mooncake 类似的内存共享系统的 RDMA API 选型。

Linux 7.2 引入 Rust zerocopy 库：4 万行 Rust 代码合入主线 内核 计算
6月15日·Phoronix / LWN.net / Miguel Ojeda 邮件列表

Linux 7.2 合并窗口的 Rust 子系统迎来体量最大的一轮更新：

  • 超 4 万行新 Rust 代码 — 由 Rust-for-Linux 维护者 Miguel Ojeda 提交，体量堪比早期一轮独立的子系统升级
  • zerocopy crate 引入 — 让内核 Rust 代码得以消除更多 unsafe 块，特别是在内存布局与类型转换场景中显著减少 UB 风险
  • AutoFDO 同步落地 — Automatic Feedback Directed Optimization 支持，使 Rust 内核代码可在编译期利用性能剖析数据优化，缩小与 C 编译路径的差距
  • 更广泛使用 — Nova DRM 驱动、Vino DisplayLink 驱动（已在 linux-next 树）、Pin 控制器、ASoC 声卡驱动均受益

此轮 Rust 改动的体量之大，是因为引入了完整的 zerocopy 库以及大量基础设施代码，预计后续 7.3 周期将看到更多 Rust 驱动的合入（如 Nova 主线化的最后阶段）。

Rust 在 Linux 内核的渗透速度正在加快 — 从最初的小型驱动试点（PIN 控制器、ASoC），到 DRM 子系统（Nova、Vino），再到系统级基础设施（zerocopy 库、AutoFDO）。这反映了三层趋势：① 内存安全是 Linux 长期维护成本的痛点，Rust 是被验证过的解药；② NVIDIA、Red Hat、Collabora 等厂商愿意投入，说明商业回报已可预期；③ 对 NPU/GPU 驱动开发者意味着 — 未来新硬件的内核驱动 Rust 比例会持续上升，掌握 Rust for Linux 已成为进阶开发的硬技能。

Linux 7.2 硬件合入潮：AMDGPU HDMI 2.1 FRL + Apple M3 初步支持 + USB4 显示 GPU
6月13-22日·Phoronix / dri-devel / Asahi Linux

Linux 7.2 合并窗口的硬件支持进入丰收期，多项备受期待的合入陆续着陆：

  • AMDGPU HDMI 2.1 FRL — 6/17 通过 DRM-Next 首次合入主线，DC Power Module 同步加入。受 HDMI Forum 许可限制多年，Valve 等合作伙伴幕后推动下绕过限制实现合规，默认禁用，需通过内核参数启用（4K@120Hz+）
  • Apple M3 初步支持 — Sven Peter 更新了 Apple SoC 设备树，Janne Grunau 提供 M3 设备树文件。这是 Asahi Linux 多年努力的成果 — 从 M1 启动到 M3 启动，主线支持迈出关键一步。iMac、MacBook Air/Pro 等机型受益
  • USB4STREAM 协议栈 — 提供 USB4 over 高速串行/PCIe 通道的设备驱动能力，为多设备外设/VR 头显铺路
  • NVIDIA Nova Rust DRM 持续推进 — Danilo Krummrich 持续提交 DRM Rust 框架改进，目标在 7.3 周期内完成 Nova 主线化
7.2 合并窗口的硬件层面有三个值得关注的信号：① AMD 用户终于有了可期的 Linux 原生 HDMI 2.1 体验，对内容创作者、HTPC 玩家意义重大；② Apple Silicon 设备的主流 Linux 支持进入新阶段，Asahi Linux 商业化（如 Fedora Asahi Remix）的潜力提升；③ USB4STREAM 暗示了未来 VR/AR、Thunderbolt 替代方案的 Linux 路径 — 这对 NPU + USB4 加速器外设的组合是利好信号。三个变更都指向同一个方向：Linux 在异构硬件上的"开箱即用"体验正在质变。
