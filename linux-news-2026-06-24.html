Linus Torvalds 亲自点评 sched_ext 源码布局"Disgusting"，内核 7.2 合入重构版本 内核 计算
6月24日·Phoronix / LKML

Linux 7.2 合并窗口期间，Linus Torvalds 对 sched_ext（可扩展调度框架） 的新增源码布局提出强烈批评，称其做法"disgusting"（恶心），随后相关代码迅速完成重构并合入主线：

事件经过：

  • 问题描述 — sched_ext 的补丁在单个 C 文件中新增了大量子调度器（sub-scheduler）相关代码，Linus 认为这种做法破坏了内核代码的组织原则，评论道："Please don't do this kind of disgusting thing… hierarchical file systems have been available since 1965."
  • 迅速响应 — 在 Linus 发帖后数小时内，sched_ext 开发者重新组织了代码结构，将子调度器支持代码拆分为合理的文件布局，新版本已合入 Linux 7.2
  • sched_ext 背景 — 该框架允许从用户空间通过 BPF 程序自定义 CPU 调度策略，已在 gaming、HPC、桌面等场景展示出显著的性能优势
  • 本次改进 — 重构后的代码持续推动子调度器（sub-scheduler）支持，使得调度策略可以进一步嵌套和组合
sched_ext 是近年来 Linux 内核调度领域最具革命性的创新之一，而 Linus 的亲自介入说明该项目已走到舞台中央：① 内核最高维护者的关注意味着 sched_ext 的代码质量正在接受最严格的审查，这对项目长期稳定性是好事；② 子调度器支持的持续推进，将使得游戏、实时计算、异构工作负载等场景的调度优化更加灵活——对 Linux 桌面和计算的未来影响深远；③ 此次"Linus 点评 → 快速修复"的流程展示了 Linux 社区健康的技术审查机制，也为其他内核子系统贡献者提供了代码组织的最佳实践参考。

Linux 7.2 在 AMD EPYC 8005 "Sorano" 上带来意外性能提升：本地网络/套接字收益显著 计算 内核
6月24日·Phoronix

Linux 7.2 合并窗口尚未关闭，Phoronix 已抢先进行了一轮 AMD EPYC 8005 Sorano 平台的基准测试，发现该版本在本地网络吞吐和套接字 I/O 方面带来了意料之外的显著性能提升：

  • 测试平台 — 2× AMD EPYC 8005 Sorano 处理器（服务器级），运行 Linux 7.1 vs 7.2-rc 开发版
  • 突出收益 — 本地（loopback）网络性能和 Unix 域套接字 I/O 有可测量的提升，具体原因仍在分析中
  • 其他改进 — Linux 7.2 包含缓存感知调度（CAS）、I/O 优化等多项通用改进，EPYC Sorano 作为 AMD 新一代服务器 APU 平台，对这些特性的响应尤为积极
  • 背景补充 — Linux 7.2 的 CAS（Cache-Aware Scheduling）由 Intel 主导合入，但在 AMD EPYC Turin/Sorano 上也观察到了收益，显示该技术具有跨厂商的通用价值

Linux 7.2 正式版预计于 2026 年 8 月初 发布，目前正处于合并窗口期（约至 6 月 29 日）。

服务器级 APU 的性能基准测试是 HPC 和云计算领域的重要参考：① EPYC Sorano 是 AMD 面向边缘和通信基础设施的新一代平台，Linux 7.2 对其的优化将直接影响电信、CDN、云游戏等场景的部署决策；② 本地网络/套接字性能的提升虽然看似"意外"，但很可能来自 7.2 中网络子系统或调度器的某次改进，值得内核网络开发者深入追溯；③ 对于在 AMD 平台上运行容器化工作负载或 AI 推理服务的团队，这一提前信号值得关注——7.2 正式发布时建议优先评估。

AMD 更新 ROCDXG：大幅改善 ROCm 在 WSL 上的体验，架构更简洁、开源、健壮 计算 GPU
6月22日·Phoronix / GitHub (ROCm)

AMD 更新了 ROCDXG 库（ROCm Driver Extension for WSL/DXG），进一步改善了 ROCm 计算栈在 Windows Subsystem for Linux (WSL2) 上的支持质量和架构整洁度：

  • ROCDXG 定位 — 让 Linux 用户空间 ROCm 库能通过 WSL 的 /dev/dxg 接口与 Windows 宿主机的 AMD GPU 驱动通信，实现"在 Windows 上原生运行 Linux AI/计算工作负载"
  • 本次改进 — 相比传统 WSL GPU 支持代码，新版 ROCDXG 架构更简洁、完全开源、异常处理更健壮，降低了 WSL 环境下 ROCm 的崩溃和内存泄漏风险
  • 使用场景 — 数据科学家可在 Windows 桌面上直接用 WSL2 运行 PyTorch/TensorFlow（ROCm 后端），无需双系统或远程 Linux 服务器
  • 支持的 GPU — 需 AMD Radeon 或 Instinct 系列 GPU，且 Windows 驱动版本与 ROCDXG 匹配

ROCDXG 是 AMD 推动 ROCm 生态"跨平台化"的关键组件，与 NVIDIA 的 CUDA-on-WSL 形成直接竞争。

AI/计算工作负载的"Windows 开发 + Linux 生产"混合场景日益普遍：① ROCDXG 的改进意味着在 Windows 笔记本/工作站上用 WSL2 做 ROCm 开发将更加稳定——对不便于安装原生 Linux 的 AI 研究者和学生尤其重要；② 相比 NVIDIA CUDA 在 WSL2 上的成熟支持，ROCm 此前一直处于追赶状态，此次更新缩小了体验差距；③ 对 AMD 而言，WSL2 支持是扩大 ROCm 市场份额的重要抓手——大量开发者工作在 Windows 上，降低进入门槛就是扩大潜在用户群。

Xfce 迎来首个 Wayland 合成器 Alpha：Xfwl4 开启桌面环境显示协议迁移新篇章 显示
6月22日·Phoronix / Xfce GitLab

专注于轻量级桌面体验的 Xfce 项目发布了其 Wayland 合成器的首个 Alpha 预览版本，标志着这个经典桌面环境正式迈出了从 X11 向 Wayland 迁移的关键一步：

  • 项目名称 — xfwl4（Xfce Wayland for wlroots 4），基于 wlroots 库构建，开发已持续约半年
  • 当前状态 — Alpha 阶段，核心 Wayland 协议支持已基本就位，但部分 Xfce 组件（面板、设置管理器）仍在适配中
  • 技术选型 — 选择 wlroots 而非 Mir 或独立实现，与 Sway、Hyprland 等现代 Wayland 合成器共享底层基础设施，有助于生态协同
  • Xfce 意义 — Xfce 是众多轻量级 Linux 发行版（如 Xubuntu、MX Linux）的默认桌面，其 Wayland 迁移进度直接影响数百万用户的显示协议升级路径

与此同期，KDE Plasma 6.8 已确认 Wayland 为默认会话，GNOME 更早已完成迁移。Xfce 的跟进意味着最后一批主流桌面环境正在告别 X11。

Xfce 的 Wayland 迁移具有特殊的生态意义：① Xfce 用户群体以"轻量、稳定、低资源占用"为核心诉求，这部分用户往往在对新技术的接受上更为保守——Xfce 迈出这一步，说明 Wayland 在非激进用户群体中的成熟度已足够；② 对显卡驱动开发者而言，Xfce 的 wlroots 选型意味着xdg-desktop-portal-wlr 和相关 Wayland 协议将成为更广泛的测试覆盖目标；③ 对 Linux 发行版维护者，Xfce Wayland 合成器的进展是制定"X11 弃用时间表"的重要参考——当 Xfce 也准备好时，整个 Linux 桌面生态的 X11 退出通道将基本清晰。

Valve 主导修复 Early AMD GCN GPU 恢复流程：老 Radeon 显卡的 Linux 稳定性将迎来改善 GPU 显示
6月21日·Phoronix / AMDGPU DRM

Valve 开源 Linux 图形团队正在改进早期 AMD GCN 架构 GPU（Radeon HD 7000 / R9 200 系列等）在 Linux 上的 GPU 挂起后恢复流程，相关补丁已提交至 amd-gfx 邮件列表：

  • 问题背景 — 部分早期 GCN GPU 在遭遇 GPU 挂起（hang）后，恢复（recovery）流程不够健壮，可能导致系统冻结或需要强制重启
  • Valve 动机 — Valve 持续投资开源 AMD Linux 驱动栈（Mesa RADV、AMDGPU 内核驱动），旨在改善 Steam Deck 及相关 AMD 硬件的游戏体验，此次改进也惠及老款硬件
  • 技术范围 — 主要涉及 AMDGPU 内核驱动中的 GFX 区块恢复顺序和超时处理逻辑，确保挂起发生后能更可靠地重置 GPU 状态
  • 硬件覆盖 — GCN 1.0 / 1.1 / 1.2 架构，对应 2012–2016 年发布的 Radeon 显卡，仍在大量旧款笔记本和台式机中使用

这也是 Valve 近期在 Linux 图形领域的系列投入之一，同期还包括 Vulkan 光追检查器（RTI）和 NVK DLSS 支持等。

Valve 对 AMD 老硬件的持续投入体现了 Linux 游戏生态的一种"长尾关怀"：① 大量 Steam 用户仍在使用 GCN 架构的老款 AMD 显卡（尤其在新兴市场和教育场景），这些硬件的 Linux 体验直接影响 Steam on Linux 的用户留存；② GPU 恢复流程的改进看似底层，但对用户而言意味着"游戏卡死 → 不必强制重启"的体验升级，是稳定性口碑的关键细节；③ 从更广阔的视角看，Valve 正在以"游戏公司"的身份系统性地补强 Linux 图形基础设施——这种来自商业实体的持续投资，比纯社区驱动更具可持续性，也是 Linux 桌面近年来取得突破的重要动力。
