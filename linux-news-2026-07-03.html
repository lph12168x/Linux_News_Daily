Intel 提交 GCC 编译器补丁：支持 x86 AI 计算扩展 ACE
计算 编译器
Phoronix · 2026-07-02 · Intel / GCC

x86 生态系统咨询组（Intel + AMD 主导）近期正式定稿了 AI Compute Extensions (ACE) v1 规范，作为 Intel AMX 的跨厂商继任者， 专为矩阵乘法和机器学习工作负载优化 x86 架构。

ACE 定义了一组矩阵乘法原语，新增 ACE 寄存器状态（含 tile 和 block scale 寄存器）、 消耗 AVX 寄存器输入的数据处理操作、ACE 与 AVX 寄存器间的数据移动操作， 以及系统管理相关状态与操作。ACE 提供 AVX 向量与 ACE tile 寄存器之间的紧密集成， 将高计算密度的 tile 处理操作与 AVX 的 comprehensive 数据处理能力相结合。

Intel 工程师已于 7 月 2 日向 GCC 邮件列表提交了首批 ACE 编译器支持补丁。 该实现部分基于 Intel 现有的 AMX-TILE 编译器代码及部分 AVX-512 元素。 预计 ACEv1 支持有望在明年（2027 年）的 GCC 17 稳定版中正式发布。

关注理由： ACE 是 x86 架构在 AI 时代的关键扩展，将直接影响未来 CPU 的矩阵计算能力， 对 GPU 加速之外的 CPU 推理场景具有重要意义。作为 AMX 的跨厂商继任者， ACE 有望获得 AMD 和 Intel 双方的处理器支持，是影响 Linux AI 计算生态的基础性进展。

Rusticl OpenCL 驱动改进硬件利用率 — Mesa 26.2 性能提升
计算 GPU
Phoronix · 2026-07-02 · Mesa / Red Hat

Red Hat 工程师 Karol Herbst 向 Mesa 26.2 提交了 Rusticl（基于 Rust 的 OpenCL 驱动）的最新优化工作， 重点改进了 workgroup 大小选择逻辑，以生成最佳的 block size。

此次改进通过更好地填充 workgroup 和 subgroup，显著提升了硬件利用率（"increase hardware utilization by a significant amount"）。 对于请求任意 global size 的应用，这一改动将带来可观的性能改善。 虽然合并请求中未提供真实工作负载的性能数据，但基于填充率的测试显示 GPU 利用率有巨大提升。

Rusticl 作为 Mesa 中基于 Rust 编写的 OpenCL 驱动，支持所有 Gallium3D 驱动后端， 是 Linux 上开源 GPGPU 计算的重要组件。该功能已合并至 Mesa 26.2 主线， 预计随 8 月稳定版一同发布。

关注理由： Rusticl 是 Linux 上摆脱专有 OpenCL 驱动（如 NVIDIA 闭源驱动）的重要开放方案， 此次硬件利用率优化直接提升了 AMD GPU 等平台上的 OpenCL 计算性能， 对科学计算、AI 推理等 GPGPU 场景有实际价值。

YSERVER 1.3 发布：Rust 编写的 X11 服务器新增 Xinerama、FreeBSD 支持
显示 X11
Phoronix · 2026-07-02 · X.Org / Rust

YSERVER 是用 Rust 编写的现代化 X11 显示服务器（借助 Claude Code AI 辅助开发）， 在发布 1.0 版本仅一个月后，已于今日发布 1.3 版本，功能快速迭代。

1.3 版本新增的主要特性包括：

  • FreeBSD 支持 — 已在 GhostBSD 上测试验证
  • musl / Alpine Linux 构建 正常工作
  • 显示热插拔（Display hotplug） 支持
  • Direct-mode VT 切换
  • Xinerama 扩展支持 — 多显示器关键扩展
  • XFIXES pointer barriers 和 XC-MISC XID 回收
  • RANDR gamma 校正 — redshift 工具可正常工作
  • 运行时 XKB 布局切换 及键盘 LED 状态驱动
  • xauth 支持

桌面环境测试方面，YSERVER 1.3 已完整测试通过 MATE、Xfce、Cinnamon 会话， 窗口管理器方面测试了 Compiz、Openbox、Awesome、Enlightenment、FVWM3、WMaker 等。 硬件测试覆盖 AMD、Intel、Snapdragon 及 Apple Silicon 显卡。

关注理由： X11 在传统 Linux 桌面中仍占重要地位，YSERVER 用 Rust 重写有望解决 C 语言 Xorg 服务器的 内存安全历史包袱。Xinerama 和多显示器支持的加入，使其更接近生产可用状态， 是显示服务器生态中值得关注的新兴项目。

RADV & RadeonSI 修复 AMD GCN 1.0/1.1 GPU 的 L2 缓存 / VM 故障
GPU 驱动
Phoronix · 2026-07-01 · Valve / AMD / Mesa

Valve Linux 图形驱动团队的 Timur Kristóf 持续推进 AMD GCN 1.0（GFX6）和 GCN 1.1（GFX7）老旧 GPU 的开源驱动支持，此次向 Mesa 提交了关键修复补丁。

根本原因是 GFX6/GFX7 硬件和固件缺陷： EVENT_WRITE_EOP 无法等待 L2 writeback，导致在 L2 仍然 dirty 时就触发 fence 信号， 进而引发 VM faults；此外，如果 VMID 在 L2 仍忙时被无效化，GFX6/GFX7 会挂起。

修复方案：对 GFX6/GFX7 的每次 submission 后，等待 shader 完成并刷新 L2 缓存。 该修复同时应用于 RADV（Vulkan）和 RadeonSI（OpenGL）两个驱动， 并额外添加了 CB/DB/meta flush。作为配套，还还原了之前针对 Hawaii 问题的 RADV 变通补丁 （因为 Linux 内核侧的 AMDGPU 修复已上游合入，Mesa 侧补丁反而会 regress Hawaii GPU 性能）。

关注理由： AMD GCN 1.0/1.1 GPU（如 Radeon HD 7000 系列、R9 280/290 等）仍有大量用户在使用， 此次修复解决了长期存在的 VM fault 稳定性问题。Valve 的持续投入也说明 Steam Deck 生态之外，老旧 AMD GPU 的 Linux 驱动维护仍是社区重点。

RISC-V RVV 向量扩展性能基准测试 — SpacemiT K3 SoC 实测
计算 SIMD
Phoronix · 2026-07-02 · RISC-V / 基准测试

Phoronix 对 SpacemiT K3 SoC（16 核，RVA23 配置，X100/A100 核心）进行了 RISC-V Vector Extension (RVV) 1.0 的性能影响专项基准测试，这是首批公开市售的支持 RVV 1.0 的 RISC-V 芯片之一。

测试环境：SpacemiT K3 Pico ITX 主板（16GB RAM），运行 Bianbu 4.0（Linux 6.18 内核，GCC 15.2）。 通过 /proc/sys/abi/riscv_v_default_allow 接口尝试运行时切换 RVV 支持， 但因 Bianbu 的 libc 构建中包含了无条件 RVV 指令，关闭该选项后系统无法执行任何二进制文件， 最终采用重新编译不含 RVV 支持的软件包的方式进行对比测试。

测试结果显示 RVV 1.0 对支持向量加速的软件包带来了显著的性能提升。 受限于 K3 SoC 当前未向 Linux 暴露任何功耗指标，本次测试未包含 RVV 开启/关闭的功耗对比数据。

关注理由： RISC-V 向量扩展（RVV）是 RISC-V 架构在 HPC 和 AI 计算领域追赶 ARM/Ax 的关键特性， 此次实测数据首次量化了 RVV 1.0 的实际性能收益，对评估 RISC-V 在 计算密集型场景的竞争力具有参考价值。
