AMDGPU HDMI 2.1 FRL 支持成功合入 Linux 7.2，历史性破冰 GPU 显示 内核
6月17日·Phoronix / DoNews / kernel.org

Linux 7.2 DRM 合并窗口首日，最受瞩目的变更尘埃落定：AMDGPU 开源驱动对 HDMI 2.1 FRL（固定速率链路）的初步支持已成功合入主线。

核心要点：

  • 多年困局终破冰 — 受 HDMI Forum 许可政策限制，AMD 开源驱动长期无法实现 HDMI 2.1。此次通过 Valve 等合作伙伴的幕后推动，以 FRL 路径绕过限制实现合规
  • 目前默认禁用 — 功能仍处于早期阶段，需通过内核参数手动启用，但已为 4K@120Hz+ 高分辨率高刷新率显示奠定基础
  • DC Power Module 同步合入 — 新的显示核心电源管理模块，使 Radeon 显卡功耗控制更接近 Windows 水平

该补丁最初于 5 月初提交，经历多轮审查后成功在 Linux 7.2 合并窗口着陆。这是 AMD 在 Linux 显示栈上的里程碑事件。

HDMI 2.1 在 Linux 上的缺失困扰了 AMD 用户近五年。FRL 支持的合并意味着 Linux 桌面/游戏用户终于可以期待原生 4K@120Hz HDR 体验，不再需要依赖 DisplayPort 或降级到 HDMI 2.0。这对 Steam Deck 生态、Linux 游戏和内容创作工作流都是直接利好。DC Power Module 的加入则改善了笔记本续航 — 两者叠加形成了桌面端和移动端的双重提升。

Vino：AI 辅助逆向工程的 Rust DRM 驱动，面向现代 DisplayLink 硬件 显示 计算
6月17日·Phoronix / dri-devel

开发者 Mike Lothian 发布了名为 "Vino" 的实验性内核驱动，专为较新的 DisplayLink USB 扩展坞打造：

  • 净室逆向工程 — 完全独立于 DisplayLink 官方闭源驱动，通过协议逆向分析实现，避免知识产权冲突
  • Rust 内核驱动 — 使用 Rust 编写，充分利用内存安全特性，并已将部分通用 Rust-for-Linux 绑定拆分为独立 RFC 补丁系列
  • AI 辅助开发 — 开发过程中使用了 Claude Code（Opus 4.8 模型） 辅助编写，是 AI 参与内核级驱动开发的前沿案例
  • 已初步测试 — 已在 Dell Universal Dock D6000（DL-3xxx 系列）上完成基础验证
现代 DisplayLink 硬件（DL-3/5/6/7xxx）在 Linux 上长期依赖树外闭源驱动和专有用户空间守护进程，体验极差。Vino 的目标是将支持推入上游，从根本上解决多屏办公场景下的 Linux 痛点。更重要的是，这展示了 Rust-for-Linux 在复杂设备驱动中的可行性，以及 AI 工具（Claude Code）参与底层系统开发的实际路径 — 两者都是 Linux 生态现代化的重要方向。

Intel Panther Lake Xe3 核显在 Linux 7.1 上获得显著性能提升 GPU 计算
6月17日·Phoronix 基准测试

继 Arc Battlemage 独显在 Linux 7.1 上表现提升后，Phoronix 对 Core Ultra X7 358H "Panther Lake" 进行了 Linux 7.0 vs 7.1 的完整对比测试：

  • Xe3 iGPU 受益明显 — Intel Arc B390（Xe3 架构）集成显卡在多项图形基准中跑出可观提升，证实 7.1 的 Battlemage 优化路径对 Xe3 同样有效
  • FRED 默认启用 — 灵活返回事件传递（FRED）在 7.1 中默认开启，为 Panther Lake 带来 CPU 侧性能增益
  • 调度器改进叠加 — 7.1 的调度器优化与 FRED 形成合力，整体平台性能水涨船高

测试平台为 MSI Prestige 14 Flip AI+ 笔记本，使用相同内核配置，仅升级内核版本以确保变量唯一。

Panther Lake 是 Intel 首款 18A 制程的移动平台，Xe3 核显性能对超薄本的游戏/内容创作能力至关重要。Linux 7.1 对其的提升说明 Intel 正在将开源图形驱动的优化节奏与硬件发布同步 — 用户不再需要等待数个内核版本才能获得完整性能。这对计划购入 Panther Lake 笔记本的 Linux 用户是重要的购前参考信号。

AI 补丁狂热冲击内核开发 — ARM64 功能被迫延期至下一周期 内核 计算
6月17日·Phoronix / LKML

Linux 7.2 合并窗口的 ARM64 架构更新中，维护者明确指出了 AI/LLM 生成补丁对开发的实际伤害：

  • "拖慢了功能开发" — ARM64 维护者在合并说明中直言，处理 AI 生成补丁的审查工作"在功能方面稍微拖慢了我们的进度"
  • 具体功能被推迟 — 部分计划在 7.2 中合并的 ARM64 特性因审查资源被 AI 补丁占用而被迫延后
  • 连锁反应扩散 — 40 年历史的 AppleTalk 协议被从内核中彻底移除，触发因素之一正是"近期 AI 生成补丁的激增"

这一现象并非孤立：AI 生成的"看起来合理但经不起推敲"的补丁正在多个子系统消耗维护者的注意力，从 ARM64 到网络协议栈无一幸免。

这不是一个"AI 将取代内核开发者"的故事 — 恰恰相反，它揭示了 AI 工具在专业领域的现实困境：低质量的 AI 生成代码正在产生真实的人力成本。对于图形/计算领域的开发者而言，ARM64 是 ARM 服务器、AI 加速器（如 NVIDIA Grace）和边缘计算平台的基础 — ARM64 功能延期意味着这些平台的 Linux 支持节奏也会受到连锁影响。这是内核社区必须正视的治理挑战。

AMD Lemonade AI Server v10.8 集成 MCP：本地 AI 推理生态加速成型 计算 NPU
6月17日·Phoronix

开源 Lemonade AI Server 发布 v10.8 版本，新增 MCP（模型上下文协议）服务器集成：

  • 全栈本地 AI — 支持 AMD Ryzen AI NPU、Radeon GPU、Instinct 加速卡和 x86_64 CPU，实现"100% 免费且私密"的本地 AI 推理
  • MCP 集成 — v10.8 允许其他应用通过标准化 MCP 协议调用 Lemonade，将其从一个独立工具升级为可嵌入工作流的 AI 基础设施
  • 跨厂商扩展 — 仅一周前发布的 v10.7 首次完成对 NVIDIA CUDA 的完整支持，打破了 AMD 生态壁垒

Lemonade 的目标是让任何拥有 AMD/NVIDIA 硬件的用户都能在本地运行大模型，无需依赖云端 API。

Lemonade 的演进代表了一股重要趋势：本地 AI 推理正在从"极客玩具"变为"桌面基础设施"。MCP 集成意味着开发者可以将本地 LLM 无缝接入 IDE、自动化脚本和日常工具链。对于注重数据隐私的 Linux 工作站/服务器用户，Lemonade + AMD/NVIDIA GPU 的组合正在成为云端 API 的可行替代方案 — 而跨厂商支持（CUDA + ROCm + NPU）的完成使这一方案摆脱了硬件绑定的局限。
