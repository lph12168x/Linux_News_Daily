Linux 7.2-rc1 发布：AMD HDMI 2.1、Intel Xe CRI、Rust zerocopy 齐亮相
Linux 内核 AMDGPU HDMI 2.1 Rust
📅 2026-06-28 🔗 WindowsForum / Phoronix 🐧 Linux 7.2-rc1

Linus Torvalds 于 2026 年 6 月 28 日发布 Linux 7.2-rc1，正式进入两周合并窗口后的公测稳定阶段。本次更新在图形与计算方向有多项实质性进展。

AMD HDMI 2.1 FRL 初步支持：AMDGPU 驱动新增 HDMI 2.1 固定速率链路（FRL）的基础支持，这是实现 HDMI 2.1 高分辨率/高刷新率传输的关键能力，缩小了 Linux 与 Windows 在消费级显示设备支持上的差距。

Intel Xe CRI 平台支持：Intel Xe 驱动新增 CRI 平台初步支持，对应 Intel 新一代显卡架构的驱动适配，提升 Intel 新架构显卡在 Linux 下的支持成熟度。

Rust zerocopy 库合入：新增 Rust `zerocopy` 库支持，用于简化内核中低开销内存操作，避免不必要内存拷贝带来的性能损耗，同时利用 Rust 的内存安全特性减少内存操作类 bug。

💡 为什么值得关注

Linux 7.2-rc1 的图形相关更新量非常大，AMD HDMI 2.1 FRL 支持是多年努力的初步成果，对使用 AMD GPU 连接 HDMI 2.1 显示器的用户意义重大；Intel CRI 平台支持则确保了新一代 Intel 显卡在 Linux 上的及时适配。Rust zerocopy 库的引入是 Linux 内核内存安全现代化的重要一步，长期来看将减少特权内存操作类漏洞。预计稳定版 Linux 7.2 将于 2026 年 8 月中旬发布。

CVE-2026-53293：AMDGPU 内核死锁漏洞修复，影响所有 AMD GPU Linux 用户
安全漏洞 AMDGPU 内核驱动
📅 2026-06-26 披露 🔗 NVD / MSRC ⚠️ CVSS 待评分

Linux 内核安全团队于 2026 年 6 月 26 日披露 CVE-2026-53293，影响 AMDGPU 驱动中 `AMDGPU_INFO_READ_MMR_REG` 接口的死锁漏洞。该漏洞源于驱动中锁操作、内存分配与用户态数据拷贝的顺序不符合内核开发规范。

根本成因：驱动在持有 GPU 重置信号量和 `mm_lock` 的状态下调用 `copy_to_user` 并向用户态拷贝数据，同时还在持锁期间执行内存分配，违反内核锁使用规范，在并发场景下可触发死锁。

影响范围：运行 Linux 且加载了 AMDGPU 驱动的设备，包括 AMD 显卡台式机/笔记本、工作站、GPU 计算服务器、SteamOS 掌机等。已修复的稳定分支版本包括 6.6.141、6.12.91、6.18.33、7.0.10 及以上。

💡 为什么值得关注

该漏洞虽为本地触发，但可导致系统完全死锁（GPU 重置路径卡住，系统无响应，只能强制重启），对多用户工作站、共享 GPU 计算节点和云游戏场景构成可用性威胁。现代 GPU 驱动已不再是单纯的"显示适配组件"，而是包含调度器、内存管理、固件交互的复杂特权组件，属于系统安全边界的重要部分。建议所有 AMD GPU Linux 用户尽快更新至已修复的内核版本。

Intel 提交 HDR over DP MST 补丁，多显示器 HDR 输出迎来原生支持
Intel i915 HDR DP MST
📅 2026-06-26 🔗 Phoronix 🐧 Linux 内核 / Mesa

Intel 工程师提交了针对 Linux 显示驱动的新补丁，支持通过 DP MST（Multi-Stream Transport）连接实现 HDR 输出，解决了多显示器环境下 HDR 能力受限的问题。

DP MST 是 DisplayPort 协议中的多显示器串联/分叉传输标准，广泛应用于通过单个 DP 接口或 Thunderbolt/USB4 扩展坞连接多台显示器的场景。此前，Linux 下通过 DP MST 连接的多显示器往往无法启用 HDR，限制了专业 HDR 工作流在多屏环境下的使用。

该补丁属于 Intel Linux 图形驱动持续完善 HDR 支持的一部分，与之前合并的 Intel HDR 基础能力形成互补，进一步完善了 Intel GPU 在 Linux 上的专业显示功能集。

💡 为什么值得关注

对于使用 Intel GPU 且通过 DP MST 扩展坞或菊花链连接多台显示器的 Linux 用户（包括大量笔记本用户），该补丁填补了多屏 HDR 支持的关键空白。HDR 在视频剪辑、图像处理和专业内容创作中已成为标配需求，该更新使 Linux 在 Intel 平台上的 HDR 支持更加完整，缩小了与 macOS/Windows 在多显示器 HDR 体验上的差距。

DXVK 3.0 发布：DXBC-SPIRV 着色器编译 + Descriptor Heaps 默认开启
DXVK Vulkan Wine / Proton
📅 2026-06-25 🔗 GitHub / Phoronix 🎮 DXVK 3.0

Philip Rebohle 宣布发布 DXVK 3.0，这是将 Direct3D 8/9/10/11 映射到 Vulkan API 的重大版本更新，面向 Wine 和 Valve Steam Play (Proton) 用户。

DXBC-SPIRV 全新着色器编译器：取代原有的传统着色器转译代码，采用基于 SSA 的 D3D SM 5.1+ 编译器，生成的 SPIR-V 代码更紧凑，着色器编译完全卸载到工作线程，显著缩短游戏加载时间。

Vulkan Descriptor Heaps 默认启用：在支持 `VK_EXT_descriptor_heap` 的 Vulkan 驱动上默认开启 Descriptor Heaps，优化内存管理效率，提升游戏运行性能。DXVK 3.0 现在要求 Vulkan 1.4 驱动级别支持。

💡 为什么值得关注

DXVK 是 Linux 游戏生态的核心组件，Steam Deck 和所有使用 Proton 运行 Windows 游戏的 Linux 用户都间接依赖它。DXBC-SPIRV 编译器的引入解决了长期存在的着色器编译卡顿问题，游戏加载时间和运行时帧稳定性均有显著改善。Descriptor Heaps 的默认开启则进一步释放了现代 Vulkan 驱动的性能潜力。对于 Linux 游戏玩家而言，这是近年来 DXVK 最重要的版本更新之一。

Nourish：基于 Vulkan 的新 Wayland 合成器发布，支持无限缩放/平移工作区
Wayland Vulkan Rust
📅 2026-06-27 🔗 Phoronix / GitHub 🖥️ Nourish Compositor

Nourish 是一款全新发布的 Wayland 合成器，其核心设计理念是"不将桌面局限于屏幕大小"，通过基于 Vulkan API 的高性能渲染和无限缩放/平移（infinite zooming/panning）能力，提供可扩展的虚拟工作区体验。

技术架构：Nourish 使用 Rust 编写的 y5 Wayland 合成器框架，底层依赖 Smithay、WGPU、Iced 等项目。渲染优先使用 Vulkan API，同时可回退至 OpenGL ES 以兼容旧硬件。已在开源 AMD、Intel 驱动及 NVIDIA 官方驱动上测试。

分数缩放支持：利用 Wayland 分数缩放协议实现缩放功能，保持画面清晰无模糊。目前提供 Fedora Linux 二进制包，源码托管于 GitHub。

💡 为什么值得关注

Wayland 合成器生态正在快速多样化，Nourish 的"无限工作区"概念为图形/创意工作者提供了类似无限画布的桌面体验，这在现有合成器（GNOME Shell、KWin、Sway、Hyprland）中较为少见。基于 Vulkan 渲染则确保了在现代 GPU 上的高性能表现。作为 Rust 编写的 Wayland 合成器，Nourish 也体现了 Linux 桌面组件向内存安全语言迁移的趋势，值得长期关注其发展。
